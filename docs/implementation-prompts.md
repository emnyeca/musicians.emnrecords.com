# 実装段階ごとのAI作業者向けプロンプト

以下を上から順に実行する。各段階のAI作業者は、最初に`AGENTS.md`、`README.md`、`docs/architecture.md`、`node_modules/next/dist/docs/`内の関係するNext.js 16.2文書、現在の作業ツリーを読むこと。関係のないユーザー変更を保持し、lint、build、追加したテストを実行し、外部環境で未確認の事項を確認済みと報告しないこと。

## 第1段階: 初期DB設計と安全な受付API

> Discord入力機能の最初の実用単位を実装してください。まだ実データがないためmigrationを追加せず、`sql/001_init.sql`と`sql/002_rls.sql`を現在の正本として再設計してください。代表者、レコード単位のロック、版番号、interaction IDによる重複防止、追記専用監査ログを追加してください。ユニットの有効な代表者は1名です。本人が変更できる公開項目と運営項目を明示的に分離してください。個別に交換できる`INTAKE_API_SECRET`で認証するサーバー側受付APIを追加し、BotへSupabaseのservice-role keyを渡さないでください。通常の正規入力は承認待ちにせず、公開プロフィール更新と監査ログ追加を同一トランザクションで即時反映してください。ロック中の更新、未知の項目、不正なURL、過大入力、古い版番号、重複interactionを拒否してください。認可、ロック、二重実行、監査失敗時のロールバックを守るテストを追加し、文書と`.env.example`を更新して検証してください。Discord投稿のクロール、Web OAuth編集、承認待ちフローは実装しないでください。

## 第2段階: Discord BotとModal入力

> Discord Botを、このリポジトリ内の独立してデプロイできるパッケージとして実装してください。必要な入力定義と検証処理だけをWeb側と共有してください。対象サーバー限定のスラッシュコマンド、メンバーロール確認、回数制限、再送防止を実装してください。Discord Modalの入力欄5個という制約を守り、必要なら複数画面に分けてください。最後に公開内容のプレビューと明示的な送信確認を表示し、受付APIへ送って即時反映してください。既存メッセージのクロール、自由投稿のDB化、BotへのDB管理権限付与、BotからのDB直接更新は行わないでください。ローカル・検証環境の設定、コマンド登録手順、解析と認可のテストを追加してください。実際のDiscord資格情報と検証用サーバーを使っていない場合、実環境で確認済みとは報告しないでください。

## 第3段階: ロック申請と限定監査通知

> 本人または運営者が対象レコードのロックを申請できるDiscord操作を実装してください。本人の申請は自分に紐づいたレコードだけを対象にし、ロックは速やかに反映してください。運営者だけがロック解除、代表者変更、運営項目変更を実行できるようにし、操作時点でロールを再確認してください。プロフィール更新、ロック、解除、失敗の結果を、秘密情報や不要な個人情報を含めず限定監査チャンネルへ通知してください。通知失敗によってDBの監査記録を失わない設計にしてください。権限外操作、重複操作、更新との競合、通知失敗をテストしてください。通常更新の承認画面や汎用的な別管理ツールは作らないでください。

## 第4段階: 復旧、代表者変更、バックアップ

> 運営者専用の代表者変更、非公開化、ロック解除、過去状態への復旧を実装してください。復旧は過去の監査情報をもとに新しい変更として反映し、復旧操作自体も監査ログへ追加してください。通常画面から物理削除できる機能は作らないでください。初期無料運用ではSupabase公式の自動バックアップやPITRを前提にせず、Supabase CLIの`db dump`、`pg_dump`、管理者向けJSON exportのいずれかを使い、追加の金銭的コストなしでプロジェクト外へ暗号化論理バックアップを保存する手順を実装してください。復元先はローカルDBまたは検証用DBとし、本番DBへ直接復元する前に内容を確認してください。秘密情報、DB接続文字列、dumpの復号鍵はリポジトリへ保存しないでください。Supabase Proの日次バックアップやPITRは、データ量、利用頻度、責任範囲が増えた場合の将来選択肢として文書化するだけでよく、初期運用の必須条件にしないでください。検証用DBへの復元を実施できる場合は実行し、できない場合は正確な手順と未確認事項を報告してください。Discordアカウント乗っ取り、Bot token流出、不正更新、監査通知失敗に対する対応手順もまとめてください。

## 第5段階: 本番前の安全性確認

> merge前の修正と本番前の安全性確認を行ってください。依存関係の脆弱性情報、Next.jsとDiscord APIの現行公式文書、RLSとDB権限、service-role keyの利用箇所、ログ、回数制限、セキュリティヘッダー、Discord署名検証、秘密情報の交換手順、エラー表示、個人情報の最小化、アクセシビリティ、スマートフォンでのModal操作を確認してください。検証環境で、本人登録、即時更新、監査記録、限定通知、重複送信、ロック、ロック解除、代表者変更、過去状態への復旧、バックアップ、復元を一通り確認してください。問題を先に修正してから文書を更新してください。最後に、Emnyecaが行う外部操作だけを短いチェックリストにし、確認済み事項と未確認事項を分けて報告してください。
