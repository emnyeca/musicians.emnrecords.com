# プロダクト方針とセキュリティ設計

## 目的と対象範囲

このプロダクトの目的は、EMN Recordsのミュージシャンを見つけやすくすることと、イベントの正確なCreditを簡単に作れるようにすることである。扱う情報は、本人が公開を希望したプロフィール情報に限る。

Credit作成画面での一時編集は、その場の出力だけに使い、名鑑DBへ書き戻さない。

立ち絵などの非公開ファイルの配布と、Discord OAuthを使ったWeb上の本人編集は対象外とする。

## 採用する情報更新の流れ

1. メンバーがEMN RecordsのDiscordサーバー内で専用コマンドを実行する。
2. Botがサーバーとメンバーロールを確認し、Discord Modalを開く。
3. Modalの入力欄は最大5個なので、必要な場合は複数画面に分け、最後に登録内容と公開される情報を本人へ表示する。
4. Botが形式を検証し、サーバー側の受付APIへ送る。既存の自己紹介チャンネルや自由投稿はクロールしない。
5. 受付APIでも同じ内容を再検証し、本人が変更できる公開項目だけをDBへ反映する。
6. 反映と同時に、変更前後・操作したDiscordユーザー・対象・時刻・結果を監査ログへ記録する。
7. 必要最小限の結果を、閲覧者を限定した監査チャンネルへ通知する。

通常の登録・更新は、運営による事前承認を必要としない。本人に紐づいた公開情報だけを変更できること、すべての変更を追跡できること、問題が起きた対象だけを速やかにロックできることを安全境界とする。

## 権限

- 一般閲覧者: `visibility = 'public'` のミュージシャンと公開リンクだけを読む。
- メンバー: 自分が代表者として紐づいた1レコードの公開プロフィール項目を登録・更新する。
- 運営者: 代表者の設定・変更、ロックと解除、非公開化、復旧を行う。
- DB書き込み処理: 信頼されたサーバー側処理だけが実行する。ブラウザ、Discord Bot、Webhook、ログへSupabaseのservice-role keyを渡さない。

ユニットやデュオも代表者を1名にする。代表者の変更とコミュニティ脱退時の権限停止は、初期運用では運営者が手動で行う。

## 本人が変更できる項目

本人が変更できる項目は、表示名、日本語名、英語名、担当、公開SNS・Webリンクなど、本人が公開するプロフィール情報だけとする。

`slug`、公開状態、確認済み状態、代表者、ロック状態、監査情報など、運営・システム管理項目は本人から変更できない。許可項目はAPI側で明示的に列挙し、入力に含まれた未知の項目は無視せず拒否する。

## ロック方針

- 通常は承認なしで即時反映する。
- 本人または運営者から申請があった場合、対象レコードだけをロックする。
- 不審な連続更新、権限不整合、認証情報流出などを検知した場合も対象レコードをロックできる。
- ロック中は本人からの更新を拒否するが、公開状態を自動変更するかは運営者が状況を見て決める。
- ロック解除、代表者変更、過去状態への復旧は運営者だけが行う。
- 公開画面から物理削除を実行する機能は作らず、通常は非公開化で対応する。

## 改ざん・誤操作への対策

- Discord側と受付API側の両方で入力を検証する。
- Discordのinteraction IDを一意キーとして扱い、再送や二重実行で同じ変更を複数回適用しない。
- DB更新と監査ログ記録を同じトランザクションで行う。監査記録に失敗した場合はプロフィールも更新しない。
- レコードの版番号を確認し、古い画面からの更新で新しい変更を上書きしない。
- Botのコマンドは対象サーバーとメンバーロールを限定し、回数制限と再送対策を行う。
- 受付APIには個別に交換できる認証情報を使う。BotへDBの管理権限を渡さない。
- 監査ログは追記専用とし、通常の更新処理から書き換え・削除できないようにする。

## バックアップと復旧

- 初期運用では金銭的コストを増やさず、Supabase Freeに自動バックアップがあると仮定しない。利用時点のSupabase FreeとProのバックアップ仕様は確認するが、未確認の無料プラン仕様を前提にしない。
- 初期の復旧方針は、追記専用の監査ログと、Supabaseプロジェクト外へ暗号化して保存する論理バックアップで成立させる。
- 論理バックアップは最低2箇所へ保存し、本番とは別の検証用DBまたはローカルDBへ復元できることを定期的に確認する。
- 通常のプロフィール復旧は、監査ログの過去状態をもとに新しい変更として反映し、復旧操作自体も監査ログへ残す。
- Supabase Proの日次バックアップやPITRは初期運用の必須条件にしない。データ量、利用頻度、運営が負う責任範囲が増え、論理バックアップだけでは復旧目標を満たせなくなった場合の将来選択肢として扱う。

## Legacyの扱い

中止した立ち絵機能は、実行可能な`legacy`サブシステムとして残さない。依存関係と攻撃対象を残さないため、コードと初期DB定義から削除し、必要な場合だけGit履歴を参照する。

現時点では実データがないため、旧構成からの移行SQLは作らない。運用開始前に、現行の初期SQLを使ってSupabase DBを作り直す。
